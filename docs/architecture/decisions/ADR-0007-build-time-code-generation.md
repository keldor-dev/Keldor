# ADR-0007: Build-Time Code Generation

| Property | Value |
|---|---|
| Status | Proposed |
| Date | 2026-07-07 |
| Decision Owner | Keldor Development Team |
| Applies To | Future Keldor build and packaging behavior |

## Context

Keldor currently uses runtime discovery to find and dot-source function files during module import.

This provides a simple development experience:

- one function per file
- predictable folders
- no manually maintained import list
- automatic public function exports

However, runtime discovery has performance costs.

Each import may require directory scans, file filtering, dot-sourcing, and export list construction.

As Keldor grows, import performance may become more important.

## Decision

Keldor may eventually use build-time code generation to create optimized module loader files during packaging.

The source repository can continue using the developer-friendly folder layout.

The packaged module can use generated import and export metadata to reduce runtime work.

## Rationale

Build-time generation provides the best of both worlds:

- simple source development
- faster runtime imports
- fewer directory scans during import
- consistent export behavior
- generated metadata that can be validated in CI

This also aligns naturally with the purpose of `Keldor.Build.PowerShell`.

## Consequences

### Positive

- Faster module imports.
- Reduced runtime file discovery.
- Build tooling can validate generated output.
- Source layout stays maintainable.
- Packaging becomes more deterministic.

### Negative

- Build process becomes more important.
- Generated files must be clearly marked.
- Contributors must know whether to edit source files or generated files.
- More build tooling is required.

## Alternatives Considered

### Continue Runtime Discovery Forever

Accepted for now, but may not be optimal as the module grows.

### Manual Static Loader

Rejected because it increases maintenance burden and invites stale imports.

### Single Monolithic Module File

Rejected because it would hurt maintainability and code navigation.

## Future Considerations

Keldor.Build.PowerShell could eventually generate:

- optimized `.psm1` loader content
- explicit dot-source statements
- public export lists
- platform-specific loading sections
- documentation metadata
- command discovery cache files

Generated files should include a clear header such as:

```powershell
# This file is generated by Keldor.Build.PowerShell. Do not edit manually.
```

This ADR should remain proposed until build-time generation is implemented and proven useful.
